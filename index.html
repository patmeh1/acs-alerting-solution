AF Group on-call critical alerts

Replacement architecture for VTEXT SMS and phone-call alerting

This solution replaces carrier email-to-SMS dependencies with Azure Monitor Action Groups for built-in SMS, voice call, email, Azure mobile push, and automation routing. It avoids using ACS SMS as a new strategic dependency because the attached ACS guidance places SMS in the retirement scope before July 31, 2028.

Recommendation: Use Azure Monitor Action Groups for critical SMS and voice notifications. Use Logic Apps or Azure Functions to normalize incoming email, scheduler, and Event Hub signals into Azure Monitor alert signals. Use Teams Phone Extensibility only if programmable call workflows, IVR, or deeper telephony automation are required.

Executive decision summary

Recommended

Azure Monitor Action Groups

Best fit for simple critical alert to SMS and phone call. It is built into Azure Monitor and supports multiple notification channels plus automation actions.

Conditional

Teams Phone Extensibility

Use when the solution needs programmable telephony, call flow control, integration with Teams Phone, or long-term ACS Call Automation support boundaries.

Avoid for new design

Standalone ACS SMS

ACS SMS is listed in the retirement scope. It can run during the transition window, but it is not a durable target architecture for this use case.

Architecture diagram

Target solution

The architecture separates event intake, alert evaluation, notification delivery, and operational tracking. This keeps the solution Microsoft-native while avoiding a direct dependency on ACS SMS or standalone ACS calling.

Log delivery, acknowledgment, incident ownership, and resolution status for audit and continuous improvement.

Implementation details

Build plan

Core Azure resources

  • Resource group for alerting components.
  • Log Analytics workspace with DCR-based custom log ingestion.
  • Azure Monitor scheduled query alerts or metric alerts.
  • Action Groups for each on-call tier or rotation.
  • Logic App Standard or Azure Function for source intake and normalization.
  • Key Vault for secrets if any non-managed identity integration is required.

Alert routing logic

  • Map severity to notification channel: Sev1 triggers voice plus SMS; lower severity can use email or Teams.
  • Map application or queue name to owning team and rotation.
  • Apply suppression windows for maintenance.
  • Apply deduplication to prevent notification storms.
  • Emit all routing decisions to Log Analytics for audit.
Source payload -> Normalize -> Write alert_event_CL -> Scheduled query alert -> Action Group -> SMS / voice / email / push

5. Sample Azure Function normalizer

This function shows the transformation logic. In production, send data to Azure Monitor using the Logs Ingestion API with a Data Collection Rule rather than the legacy HTTP Data Collector API.

import json
import logging
import azure.functions as func

app = func.FunctionApp()

@app.event_hub_message_trigger(
    arg_name="event",
    event_hub_name="%EVENTHUB_NAME%",
    connection="EVENTHUB_CONNECTION"
)
def normalize_alert(event: func.EventHubEvent):
    payload = json.loads(event.get_body().decode("utf-8"))

    normalized = {
        "Source": payload.get("source", "unknown"),
        "Application": payload.get("application", "unknown"),
        "Severity": payload.get("severity", "Sev3"),
        "Summary": payload.get("summary", "Alert received"),
        "Details": payload.get("details", ""),
        "IncidentId": payload.get("incidentId", ""),
        "OwnerGroup": payload.get("ownerGroup", "Unassigned"),
        "DedupeKey": payload.get("dedupeKey", ""),
    }

    if normalized["Severity"] == "Sev1":
        logging.error("CRITICAL_ALERT %s", json.dumps(normalized))
    else:
        logging.warning("ALERT %s", json.dumps(normalized))

6. Create an Action Group for voice and SMS

Replace the sample phone number and email with approved on-call contacts. Test every recipient before production use.

az monitor action-group create \
  --resource-group "$RG" \
  --name "$ACTION_GROUP" \
  --short-name "AFSEV1" \
  --action email oncall-email oncall@example.com \
  --action sms oncall-sms 1 5550101234 \
  --action voice oncall-voice 1 5550101234

7. Create a scheduled query alert

The final KQL depends on the chosen ingestion table. This sample illustrates the target behavior: fire only for recent Sev1 normalized alerts.

let window = 5m;
AlertEvents_CL
| where TimeGenerated > ago(window)
| where Severity_s == "Sev1"
| summarize Count=count() by DedupeKey_s, Application_s, Summary_s
LAW_ID=$(az monitor log-analytics workspace show \
  --resource-group "$RG" \
  --workspace-name "$LAW" \
  --query id -o tsv)

AG_ID=$(az monitor action-group show \
  --resource-group "$RG" \
  --name "$ACTION_GROUP" \
  --query id -o tsv)

az monitor scheduled-query create \
  --resource-group "$RG" \
  --name "$ALERT_RULE" \
  --scopes "$LAW_ID" \
  --condition "count 'AlertEvents_CL | where TimeGenerated > ago(5m) | where Severity_s == \"Sev1\"' > 0" \
  --condition-query AlertEvents_CL \
  --evaluation-frequency 5m \
  --window-size 5m \
  --severity 1 \
  --action-groups "$AG_ID" \
  --description "AF Group Sev1 critical alert notification"

  • Action Groups per team or rotation.
  • Alert processing rules for maintenance windows.
  • Dashboards for delivery and incident metrics.
  • Quarterly tests of voice, SMS, email, and fallback paths.

If AF Group needs programmable calls, IVR, recording, transcription, or call-flow logic, evaluate Teams Phone Extensibility with ACS Call Automation under the supported Teams-aligned scenarios.

Open questions

What is missing or should be confirmed

Question Why it matters Decision impact
Which countries and carriers must receive SMS and voice calls? Azure Monitor Action Group SMS and voice availability varies by country/region. May require Teams Phone Extensibility or Marketplace partner fallback for unsupported destinations.
Where is the source of truth for the on-call rotation? Static Action Groups work for small teams, but dynamic rotations need automation or integration with a scheduling system. Determines whether to manage Action Groups manually, update them through automation, or integrate an on-call platform.
Do alerts require acknowledgment and escalation if nobody responds? Action Groups notify; they are not a full incident acknowledgment workflow by themselves. May require ITSM integration, Teams workflow, Logic App escalation, or an on-call product.
What are the expected alert volumes and peak storm scenarios? Cost and alert fatigue depend on frequency, retries, deduplication, and severity filtering. Determines dedupe design, severity thresholds, and suppression strategy.
Are toll-free numbers or registered business caller IDs mandatory? The screenshot mentions toll-free and registered business numbers, but Action Group notifications use Microsoft-managed notification delivery rather than a custom ACS caller ID. If branded caller ID is mandatory, evaluate Teams Phone Extensibility or a communications partner.
Is this only for Azure workloads, or also on-premises and third-party jobs? Non-Azure sources need explicit ingestion through Event Hub, Logic Apps, Functions, or the Logs Ingestion API. Determines connectivity, identity, firewall, and monitoring design.

Risks and decisions

Key decisions to make

Risks

  • SMS and voice notification availability varies by country and recipient configuration.
  • Notification storms can occur if deduplication and suppression are not designed up front.
  • Existing email-to-SMS behavior may have masked missing incident ownership metadata.
  • ACS retirement scope makes direct ACS SMS unsuitable as a strategic replacement.

Decisions

  • Which alerts are Sev1 and require voice calls?
  • Where should the source of truth for on-call rotation live?
  • Should Teams or ITSM be the system of record for acknowledgment?
  • Is simple notification enough, or is programmable telephony required?
Guiding principle: Use the simplest built-in Azure capability that reliably wakes up the correct on-call owner. Add Teams Phone Extensibility only when business requirements exceed what Azure Monitor Action Groups can provide.

Run a two-week pilot with one production support mailbox, one scheduler failure event, one Event Hub test message, and one Action Group containing current on-call contacts. Success criteria: SMS and voice delivery confirmed, no duplicate storms, audit trail captured, and support runbook approved.
