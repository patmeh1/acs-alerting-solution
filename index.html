AF Group on-call critical alerts

Replacement architecture for VTEXT SMS and phone-call alerting

This solution replaces carrier email-to-SMS dependencies with Azure Monitor Action Groups for built-in SMS, voice call, email, Azure mobile push, and automation routing. It avoids using ACS SMS as a new strategic dependency because the attached ACS guidance places SMS in the retirement scope before July 31, 2028.

Recommendation: Use Azure Monitor Action Groups for critical SMS and voice notifications. Use Logic Apps or Azure Functions to normalize incoming email, scheduler, and Event Hub signals into Azure Monitor alert signals. Use Teams Phone Extensibility only if programmable call workflows, IVR, or deeper telephony automation are required.

Executive decision summary

Recommended

Azure Monitor Action Groups

Best fit for simple critical alert to SMS and phone call. It is built into Azure Monitor and supports multiple notification channels plus automation actions.

Conditional

Teams Phone Extensibility

Use when the solution needs programmable telephony, call flow control, integration with Teams Phone, or long-term ACS Call Automation support boundaries.

Avoid for new design

Standalone ACS SMS

ACS SMS is listed in the retirement scope. It can run during the transition window, but it is not a durable target architecture for this use case.

Architecture diagram

Target solution

The architecture separates event intake, alert evaluation, notification delivery, and operational tracking. This keeps the solution Microsoft-native while avoiding a direct dependency on ACS SMS or standalone ACS calling.

Implementation details

Build plan

Core Azure resources

  • Resource group for alerting components.
  • Log Analytics workspace with DCR-based custom log ingestion.
  • Azure Monitor scheduled query alerts or metric alerts.
  • Action Groups for each on-call tier or rotation.
  • Logic App Standard or Azure Function for source intake and normalization.
  • Key Vault for secrets if any non-managed identity integration is required.

Alert routing logic

  • Map severity to notification channel: Sev1 triggers voice plus SMS; lower severity can use email or Teams.
  • Map application or queue name to owning team and rotation.
  • Apply suppression windows for maintenance.
  • Apply deduplication to prevent notification storms.
  • Emit all routing decisions to Log Analytics for audit.
Source payload -> Normalize -> Write alert_event_CL -> Scheduled query alert -> Action Group -> SMS / voice / email / push

Short-term

Stabilize the existing alerting gap

Week 1

Proof of value

Create a pilot Action Group with two or three test recipients. Validate SMS and voice delivery for target countries and carriers.

Weeks 2-3

Source integration

Implement Logic App or Function intake for the highest-value monitored mailbox and scheduler alert source. Emit normalized events into Azure Monitor.

Weeks 4-6

Operational rollout

Move Sev1 and Sev2 alerts to the new route, run parallel with existing VTEXT where available, and document support procedures.

Long-term

Durable operating model

Preferred long-term path

Standardize production support alerting on Azure Monitor Action Groups and alert processing rules. Use Azure Policy and templates to create consistent alerting patterns across applications.

  • Action Groups per team or rotation.
  • Alert processing rules for maintenance windows.
  • Dashboards for delivery and incident metrics.
  • Quarterly tests of voice, SMS, email, and fallback paths.

Telephony-heavy path

If AF Group needs programmable calls, IVR, recording, transcription, or call-flow logic, evaluate Teams Phone Extensibility with ACS Call Automation under the supported Teams-aligned scenarios.

  • Confirm Teams Phone licensing and connectivity.
  • Validate supported scenarios with Microsoft guidance.
  • Avoid unsupported standalone ACS usage after July 31, 2028.

Risks and decisions

Key decisions to make

Risks

  • SMS and voice notification availability varies by country and recipient configuration.
  • Notification storms can occur if deduplication and suppression are not designed up front.
  • Existing email-to-SMS behavior may have masked missing incident ownership metadata.
  • ACS retirement scope makes direct ACS SMS unsuitable as a strategic replacement.

Decisions

  • Which alerts are Sev1 and require voice calls?
  • Where should the source of truth for on-call rotation live?
  • Should Teams or ITSM be the system of record for acknowledgment?
  • Is simple notification enough, or is programmable telephony required?
Guiding principle: Use the simplest built-in Azure capability that reliably wakes up the correct on-call owner. Add Teams Phone Extensibility only when business requirements exceed what Azure Monitor Action Groups can provide.

Proposed next step

Run a two-week pilot with one production support mailbox, one scheduler failure event, one Event Hub test message, and one Action Group containing current on-call contacts. Success criteria: SMS and voice delivery confirmed, no duplicate storms, audit trail captured, and support runbook approved.
