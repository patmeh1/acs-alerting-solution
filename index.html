On-call critical alerts

Replacement architecture for carrier email-to-SMS and phone-call alerting

This solution replaces carrier email-to-SMS dependencies with Azure Monitor Action Groups for built-in SMS, voice call, email, Azure mobile push, and automation routing. It avoids anchoring on any standalone SMS or voice CPaaS product without a published long-term support roadmap — the Azure Communication Services SMS path is a concrete example currently in its announced retirement window (full retirement 31 July 2028).

Recommendation: Use Azure Monitor Action Groups for critical SMS and voice notifications. Use Logic Apps or Azure Functions to normalize incoming email, scheduler, and Event Hub signals into Azure Monitor alert signals. Use Teams Phone Extensibility only if programmable call workflows, IVR, or deeper telephony automation are required.
ACS retirement context. Microsoft announced Azure Communication Services retirement on 22 July 2026 with full retirement on 31 July 2028 — see the retirement and breaking changes guide. ACS SMS sits inside that scope, so it is not a strategic target for a net-new on-call alerting build. Azure Monitor Action Groups use Microsoft's own notification network and are not part of the ACS retirement.

Executive decision summary

Recommended

Azure Monitor Action Groups

Best fit for simple critical alert to SMS and phone call. It is built into Azure Monitor and supports multiple notification channels plus automation actions.

Conditional

Teams Phone with Direct Routing

Use when the solution needs programmable Teams telephony, branded caller ID, IVR, or call recording. Pair with Logic Apps for orchestration. Avoid taking a hard dependency on telephony products that do not have a published long-term support roadmap.

Avoid for new design

Standalone SMS/voice CPaaS as a strategic dependency

Standalone SMS/voice CPaaS products without a published long-term support roadmap (including the ACS SMS path that is currently in its retirement window) should not anchor a new on-call solution. Use Azure Monitor Action Groups for first-party notification and an Azure Marketplace partner only as a regional fallback if needed.

Updated use case

What changes with the added detail

An earlier proposal had Outlook, monitoring systems, or batch jobs flow through Logic Apps and Azure Functions out to a standalone SMS/voice CPaaS service. The event-processing pattern is sound, but the outbound telephony portion should be revised. Standalone SMS/voice services without a published long-term support roadmap (which includes the ACS SMS path currently in its retirement window) are not durable targets for a net-new build. Replace that leg with Azure Monitor Action Groups for first-party SMS, voice, email, and push.

Original pattern from the added detail

Outlook / Monitoring System / Batch Job
  -> Azure Logic App
  -> Azure Function
  -> External SMS/voice CPaaS
  -> Voice Call / SMS
  -> Event Grid status tracking

Revised Microsoft-native pattern

Outlook / Monitoring System / Batch Job / Event Hub
  -> Logic App or Azure Function
  -> Log Analytics or Application Insights
  -> Azure Monitor Alert Rule
  -> Action Group
  -> Voice call + SMS + email + push + ITSM/webhook
Scenario Primary channel Why
Critical failures, outages, Sev1 production incidents Voice call plus SMS Voice is more likely to wake the on-call engineer; SMS provides a written summary.
Escalations or after-hours alerts Voice call plus SMS plus email Escalation requires redundancy across channels.
Immediate attention required but not full outage SMS plus Azure mobile push Fast notification without interruptive phone call unless unacknowledged.
Simple notifications or FYI alerts Email or Teams notification Reduces fatigue and avoids unnecessary SMS/voice spend.

Architecture diagram

Target solution

The architecture separates event intake, alert evaluation, notification delivery, and operational tracking. This keeps the solution Microsoft-native while avoiding a direct dependency on any standalone SMS or voice CPaaS product whose long-term support is not guaranteed.

Page view: end-to-end flow

1

Detect

Monitor production support mailbox, batch scheduler failures, application alerts, and Event Hub messages.

2

Normalize

Logic App or Function parses source payloads, assigns severity, maps service ownership, and removes duplicate alerts.

3

Evaluate

Azure Monitor alert rule evaluates custom logs, metrics, or App Insights events and applies alert processing rules.

4

Notify

Action Group sends SMS, voice call, email, push, and optional webhook or ITSM automation.

5

Track

Log delivery, acknowledgment, incident ownership, and resolution status for audit and continuous improvement.

Implementation details

Build plan

Core Azure resources

  • Resource group for alerting components.
  • Log Analytics workspace with DCR-based custom log ingestion.
  • Azure Monitor scheduled query alerts or metric alerts.
  • Action Groups for each on-call tier or rotation.
  • Logic App Standard or Azure Function for source intake and normalization.
  • Key Vault for secrets if any non-managed identity integration is required.

Alert routing logic

  • Map severity to notification channel: Sev1 triggers voice plus SMS; lower severity can use email or Teams.
  • Map application or queue name to owning team and rotation.
  • Apply suppression windows for maintenance.
  • Apply deduplication to prevent notification storms.
  • Emit all routing decisions to Log Analytics for audit.
Source payload -> Normalize -> Write alert_event_CL -> Scheduled query alert -> Action Group -> SMS / voice / email / push

5. Sample Azure Function normalizer

This function shows the transformation logic. In production, send data to Azure Monitor using the Logs Ingestion API with a Data Collection Rule rather than the legacy HTTP Data Collector API.

import json
import logging
import azure.functions as func

app = func.FunctionApp()

@app.event_hub_message_trigger(
    arg_name="event",
    event_hub_name="%EVENTHUB_NAME%",
    connection="EVENTHUB_CONNECTION"
)
def normalize_alert(event: func.EventHubEvent):
    payload = json.loads(event.get_body().decode("utf-8"))

    normalized = {
        "Source": payload.get("source", "unknown"),
        "Application": payload.get("application", "unknown"),
        "Severity": payload.get("severity", "Sev3"),
        "Summary": payload.get("summary", "Alert received"),
        "Details": payload.get("details", ""),
        "IncidentId": payload.get("incidentId", ""),
        "OwnerGroup": payload.get("ownerGroup", "Unassigned"),
        "DedupeKey": payload.get("dedupeKey", ""),
    }

    if normalized["Severity"] == "Sev1":
        logging.error("CRITICAL_ALERT %s", json.dumps(normalized))
    else:
        logging.warning("ALERT %s", json.dumps(normalized))

6. Create an Action Group for voice and SMS

Replace the sample phone number and email with approved on-call contacts. Test every recipient before production use.

az monitor action-group create \
  --resource-group "$RG" \
  --name "$ACTION_GROUP" \
  --short-name "ONCALL" \
  --action email oncall-email oncall@example.com \
  --action sms oncall-sms 1 5550101234 \
  --action voice oncall-voice 1 5550101234

7. Create a scheduled query alert

The final KQL depends on the chosen ingestion table. This sample illustrates the target behavior: fire only for recent Sev1 normalized alerts.

let window = 5m;
AlertEvents_CL
| where TimeGenerated > ago(window)
| where Severity_s == "Sev1"
| summarize Count=count() by DedupeKey_s, Application_s, Summary_s
LAW_ID=$(az monitor log-analytics workspace show \
  --resource-group "$RG" \
  --workspace-name "$LAW" \
  --query id -o tsv)

AG_ID=$(az monitor action-group show \
  --resource-group "$RG" \
  --name "$ACTION_GROUP" \
  --query id -o tsv)

az monitor scheduled-query create \
  --resource-group "$RG" \
  --name "$ALERT_RULE" \
  --scopes "$LAW_ID" \
  --condition "count 'AlertEvents_CL | where TimeGenerated > ago(5m) | where Severity_s == \"Sev1\"' > 0" \
  --condition-query AlertEvents_CL \
  --evaluation-frequency 5m \
  --window-size 5m \
  --severity 1 \
  --action-groups "$AG_ID" \
  --description "the customer Sev1 critical alert notification"

8. Logic App option for monitored mailbox intake

Trigger: When a new email arrives in shared mailbox
Condition: Subject/body contains "CRITICAL", "FAILED", or configured job names
Action: Compose normalized alert JSON
Action: Send event to Event Hub or call HTTP-triggered Function
Action: Write audit record to Log Analytics
Action: Optional Teams message to production support channel

9. Validate the pilot

  • Send a synthetic Event Hub message for a Sev1 batch failure.
  • Confirm Function logs show one normalized alert event.
  • Confirm Azure Monitor alert fires within the expected evaluation window.
  • Confirm Action Group sends voice, SMS, email, and push notifications.
  • Confirm duplicate payloads do not create repeated phone calls during the dedupe window.
  • Confirm a planned maintenance processing rule suppresses notifications when expected.

One-shot deployable pilot

  • Log Analytics workspace with a daily ingestion cap to control cost.
  • Workspace-based Application Insights for Function telemetry.

  • Scheduled query alert rule that fires on Sev1 records in AlertEvents_CL.
  • System-assigned managed identity on the Function for future least-privilege wiring.

Short-term

Stabilize the existing alerting gap

Week 1

Proof of value

Create a pilot Action Group with two or three test recipients. Validate SMS and voice delivery for target countries and carriers.

Weeks 2-3

Source integration

Implement Logic App or Function intake for the highest-value monitored mailbox and scheduler alert source. Emit normalized events into Azure Monitor.

Weeks 4-6

Operational rollout

Move Sev1 and Sev2 alerts to the new route, run parallel with existing carrier email-to-SMS where available, and document support procedures.

Long-term

Durable operating model

Preferred long-term path

Standardize production support alerting on Azure Monitor Action Groups and alert processing rules. Use Azure Policy and templates to create consistent alerting patterns across applications.

  • Action Groups per team or rotation.
  • Alert processing rules for maintenance windows.
  • Dashboards for delivery and incident metrics.
  • Quarterly tests of voice, SMS, email, and fallback paths.

Telephony-heavy path

If the customer needs programmable calls, IVR, recording, transcription, or call-flow logic, evaluate Teams Phone with Direct Routing or an Azure Marketplace partner CPaaS with a published long-term support roadmap. Wrap the telephony product in a thin Logic App or Function so it can be swapped without changing the alert ingestion layer.

  • Confirm Teams Phone licensing and SBC connectivity if you go the Direct Routing route.
  • For Marketplace partners, require contractual SLAs on SMS and voice delivery for every country of operation.
  • Do not anchor on any product that lacks a published long-term support commitment.

Open questions

What is missing or should be confirmed

Question Why it matters Decision impact
Which countries and carriers must receive SMS and voice calls? Azure Monitor Action Group SMS and voice availability varies by country/region. May require Teams Phone Extensibility or Marketplace partner fallback for unsupported destinations.
Where is the source of truth for the on-call rotation? Static Action Groups work for small teams, but dynamic rotations need automation or integration with a scheduling system. Determines whether to manage Action Groups manually, update them through automation, or integrate an on-call platform.
Do alerts require acknowledgment and escalation if nobody responds? Action Groups notify; they are not a full incident acknowledgment workflow by themselves. May require ITSM integration, Teams workflow, Logic App escalation, or an on-call product.
What are the expected alert volumes and peak storm scenarios? Cost and alert fatigue depend on frequency, retries, deduplication, and severity filtering. Determines dedupe design, severity thresholds, and suppression strategy.
Are toll-free numbers or registered business caller IDs mandatory? Action Group notifications use Microsoft-managed notification delivery rather than a custom branded caller ID. If branded caller ID is mandatory, evaluate Teams Phone with Direct Routing or a partner CPaaS service from Azure Marketplace.
Is this only for Azure workloads, or also on-premises and third-party jobs? Non-Azure sources need explicit ingestion through Event Hub, Logic Apps, Functions, or the Logs Ingestion API. Determines connectivity, identity, firewall, and monitoring design.

Risks and decisions

Key decisions to make

Risks

  • SMS and voice notification availability varies by country and recipient configuration.
  • Notification storms can occur if deduplication and suppression are not designed up front.
  • Existing email-to-SMS behavior may have masked missing incident ownership metadata.
  • Any SMS or voice product without a published long-term support roadmap (including the ACS SMS path that is currently in its retirement window) is unsuitable as a strategic replacement.

Decisions

  • Which alerts are Sev1 and require voice calls?
  • Where should the source of truth for on-call rotation live?
  • Should Teams or ITSM be the system of record for acknowledgment?
  • Is simple notification enough, or is programmable telephony required?
Guiding principle: Use the simplest built-in Azure capability that reliably wakes up the correct on-call owner. Add Teams Phone Extensibility only when business requirements exceed what Azure Monitor Action Groups can provide.
